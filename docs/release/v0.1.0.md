# Codex Harness v0.1.0 Draft Release Notes

Status: draft only. This note is for the first manual GitHub Release and does not mean the release has been published.

## Overview

Codex Harness v0.1.0 is an early-stage gated workflow plugin for Codex CLI. It is meant for experimenting with deliberate agent workflows: planning, exact approval, scoped implementation, clean-context review, repair planning, repair approval, and completion reporting.

Harness is not a sandbox, security boundary, or production security product. It does not replace least-privilege permissions, project security controls, code review, automated verification, or human judgment.

## Highlights

- Adds the Harness workflow for separating Plan, implementation, Review, repair, and Completion stages.
- Supports command-like activation when `$harness` starts a non-quoted, non-fenced prompt line, or when `use $harness` starts such a line.
- Includes minimal hooks for Harness artifact shape checks and obvious dangerous command pattern blocking.
- Documents the denylist guard limitation: recognized dangerous patterns can be blocked, but unrecognized inputs can fail open by design.
- Produces the plugin bundle as `dist/harness.zip` through `make package`.

## Verification Expectations

Before publishing the manual release, run the repository verification already covered by CI:

- `make test`
- `make package`
- inspect `dist/harness.zip`, for example with `unzip -l dist/harness.zip`

The archive should remain a loadable Codex plugin bundle and should not include local build output, ignored local files, `.git/`, `__MACOSX/`, `__pycache__/`, `*.pyc`, `.DS_Store`, or nested `dist/` paths.

## Manual Release Asset Guidance

For the first GitHub Release, create the `v0.1.0` tag manually, create the GitHub Release, and upload `dist/harness.zip` as the release asset. Mark the release as pre-release if that better reflects the early-stage status.

Do not describe this release as published until the GitHub Release exists and the asset has been uploaded.
